# AI Sound Library Analyzer - Data Model

Version: v0.1

---

# 1. Purpose

This document defines the internal data representation used by AI Sound Library Analyzer.

The objective is to transform raw sound libraries into structured machine-readable information.

The data model must support:

* preset analysis
* audio analysis
* MIDI analysis
* dataset creation
* AI training
* sound generation

---

# 2. Core Concept

A sound is represented as a combination of:

```
Sound

 |
 +-- Metadata
 |
 +-- Source Information
 |
 +-- Synth Parameters
 |
 +-- Audio Features
 |
 +-- Musical Features
 |
 +-- Semantic Tags
 |
 +-- AI Embedding
```

---

# 3. Main Entity: Sound Object

Every analyzed item becomes a Sound Object.

Example:

```json
{
    "id": "serum_bass_001",

    "source": {
        "library": "Techno Bass Collection",
        "file": "DeepBass.fxp"
    },

    "type": "preset",

    "plugin": {
        "name": "Serum",
        "vendor": "Xfer Records"
    },

    "classification": {
        "category": "bass",
        "style": "techno",
        "character": [
            "dark",
            "aggressive",
            "analog"
        ]
    }
}
```

---

# 4. Metadata Layer

General information about the object.

Schema:

```json
{
    "metadata": {

        "id": "",
        "filename": "",
        "extension": "",
        "created_at": "",

        "source_library": "",

        "hash": ""

    }
}
```

The hash allows:

* duplicate detection
* dataset cleaning
* version tracking

---

# 5. Source Layer

Describes where the sound comes from.

Example:

```json
{
    "source": {

        "library": "Afterlife Inspired",

        "vendor": "Unknown",

        "path":
        "libraries/Afterlife/Bass/Bass01.fxp"

    }
}
```

The final dataset will contain:

```
dataset.json

|

+-- sounds[]

      |
      +-- sound object

      |
      +-- features

      |
      +-- labels

      |
      +-- embedding
```

Example:

```json
{

"sounds":[

{

"id":"001",

"label":"bass",

"features":[

0.2,

0.8,

0.4

]

}

]

}
```

---

# 16. Design Rules

## Rule 1

Never store only descriptions.

Always prefer numerical features.

Bad:

```json
"warm bass"
```

Good:

```json
"warmth":0.85
```

---

## Rule 2

Raw data and AI data must stay separated.

Example:

```
raw_analysis.json

        |

        v

dataset.json
```

---

## Rule 3

Every feature must be optional.

Different plugins expose different parameters.
